# Professional Web Design

Every web interface you produce must look like it was crafted by an experienced designer, not generated by AI. This skill ensures that happens consistently across all project types and tech stacks.

## Core Philosophy

**No AI slop.** The #1 failure mode is producing generic, cookie-cutter interfaces that all look the same. Every project deserves a unique visual identity that matches its purpose.

Before writing any code, spend 30 seconds thinking about:
1. **Who is this for?** (developer tool → dense & efficient, consumer product → inviting & clear, creative portfolio → expressive & bold)
2. **What's the vibe?** Pick ONE clear direction and commit fully. Timid design is bad design.
3. **What makes it memorable?** Identify one signature element (a distinctive color, an animation, a layout choice) that someone would remember.

## Design Decision Framework

Based on the project type, select an aesthetic approach:

| Project Type | Default Direction | Key Priorities |
|---|---|---|
| Landing page / marketing | Bold, conversion-focused | Hero impact, clear CTA hierarchy, social proof layout |
| SaaS / dashboard | Clean, information-dense | Data readability, consistent spacing, subtle interactions |
| Portfolio / personal | Expressive, distinctive | Personality, memorable layout, creative typography |
| E-shop | Trust-building, scannable | Product focus, clear pricing, smooth browse flow |
| Blog / content | Readable, elegant | Typography-first, comfortable reading width, whitespace |
| Admin / internal tool | Efficient, structured | Dense but organized, clear status indicators, fast navigation |

These are starting points — override them when the project context demands it.

## Typography Rules

Typography is the single biggest differentiator between amateur and professional design.

**Always use Google Fonts.** Import them properly. NEVER fall back to system fonts as the primary choice.

**Font selection strategy:**
- Read `references/typography.md` for curated font pairings organized by aesthetic
- Pick a DISPLAY font (headings) + BODY font (text) that complement each other
- The display font carries the personality; the body font carries readability
- NEVER use: Inter, Roboto, Arial, system-ui, or sans-serif as design choices (they're fine as fallbacks only)

**Typographic scale:** Use a consistent scale. A good default is 1.25 ratio:
- xs: 0.75rem, sm: 0.875rem, base: 1rem, lg: 1.25rem, xl: 1.563rem, 2xl: 1.953rem, 3xl: 2.441rem, 4xl: 3.052rem

**Line heights:** Body text 1.5–1.7, headings 1.1–1.3. Never leave line-height at default.

**Letter spacing:** Tighten headings (-0.02em to -0.04em), slightly open small caps and labels (+0.05em to +0.1em).

## Color System

**Build a proper palette, not random hex values.**

Every project needs:
- **Primary**: The brand/accent color (1 color, 2-3 shades)
- **Neutral**: Background and text shades (at minimum: bg, surface, border, text-muted, text-primary)
- **Semantic**: Success, warning, error, info (only if the UI needs status indicators)
- **Accent** (optional): A secondary color for variety — use sparingly

**Use CSS custom properties.** Define your palette in `:root` and reference everywhere:

```css
:root {
  --color-bg: #0a0a0b;
  --color-surface: #141416;
  --color-border: #2a2a2e;
  --color-text: #e8e8ed;
  --color-text-muted: #8a8a96;
  --color-primary: #6366f1;
  --color-primary-hover: #818cf8;
}
```

**Color contrast:** Ensure WCAG AA minimum (4.5:1 for body text, 3:1 for large text). When in doubt, increase contrast.

**Dark vs Light:** Choose based on context. Don't default to dark for everything. Content-heavy sites often work better light. Tools and dashboards can go either way. Let the project's audience guide you.

## Layout & Spacing

**Use a spacing system.** Pick a base unit (4px or 8px) and multiply. Never use arbitrary pixel values.

Good spacing scale (base 4px): 4, 8, 12, 16, 24, 32, 48, 64, 96, 128

**Max content width:** Always constrain. Body text: 65-75ch. Full layouts: 1200-1440px. Never let content stretch to full viewport on wide screens.

**CSS Grid > Flexbox** for page-level layouts. Flexbox for component-level alignment.

**Whitespace is a feature.** Generous padding inside cards/sections. Don't cram elements together. The space between elements communicates hierarchy as much as size does.

**Responsive design is mandatory.** Every layout must work on mobile. Use `clamp()` for fluid typography and spacing. Design mobile-first when practical.

## Visual Depth & Polish

These details separate professional from amateur:

**Shadows:** Use layered, subtle shadows — not single harsh box-shadows.
```css
/* Good: layered shadow */
box-shadow: 0 1px 2px rgba(0,0,0,0.06), 0 4px 12px rgba(0,0,0,0.08);

/* Bad: harsh single shadow */
box-shadow: 0 4px 8px rgba(0,0,0,0.3);
```

**Border radius:** Be consistent. Pick a radius scale and stick to it (e.g., sm: 6px, md: 10px, lg: 16px, full: 9999px).

**Borders:** Prefer subtle borders (1px, low-opacity) over heavy outlines. Use border color that's close to the background.

**Backgrounds:** Add subtle texture or gradient. A slight gradient (even 2% opacity shift) adds depth. Noise texture overlays (via SVG filter or pseudo-element) add sophistication.

**Hover/focus states:** Every interactive element needs them. Transitions should be 150-250ms with ease-out. Never instant, never slow.

```css
transition: all 0.2s ease-out;
```

## Animations & Motion

Motion should be purposeful, not decorative.

**Page load:** Stagger element entry with subtle fade-up animations. 
```css
@keyframes fadeUp {
  from { opacity: 0; transform: translateY(12px); }
  to { opacity: 1; transform: translateY(0); }
}
.animate-in {
  animation: fadeUp 0.5s ease-out forwards;
}
/* Stagger with animation-delay on children */
```

**Scroll animations:** Use `IntersectionObserver` for scroll-triggered reveals. Avoid heavy scroll-jacking.

**Micro-interactions:** Button press feedback, input focus effects, toggle animations. Small but impactful.

**Performance:** Animate only `transform` and `opacity`. Never animate `width`, `height`, `top`, `left`, or `margin`.

## Component Patterns

Read `references/components.md` for detailed component patterns. Key principles:

**Buttons:** Clear hierarchy — primary (filled), secondary (outlined), ghost (text-only). Adequate padding (min 12px vertical, 24px horizontal). Don't make them too small on touch devices (min 44px tap target).

**Cards:** Consistent internal spacing. Clear visual boundary (border or shadow, not both). Content hierarchy within the card.

**Navigation:** Sticky where appropriate. Clear active state. Mobile: hamburger or bottom nav, never a horizontal scroll of links.

**Forms:** Labels above inputs. Adequate input height (min 44px). Clear focus states. Inline validation with helpful messages.

**Tables:** Alternating row colors or clear row separators. Sticky headers for long tables. Right-align numbers. Adequate cell padding.

## Tech Stack Adaptations

Apply these principles regardless of framework:

**Plain HTML/CSS/JS:** Use CSS custom properties, semantic HTML5 elements, CSS Grid/Flexbox. Import Google Fonts via `<link>`.

**React + Tailwind:** Use Tailwind utility classes but extend the config with custom colors, fonts, and spacing. Don't settle for Tailwind defaults — customize the theme.

**Next.js:** Use `next/font` for Google Fonts. Use CSS Modules or Tailwind. Leverage `next/image` for optimized images.

**Vue/Nuxt:** Scoped styles with CSS custom properties. Same design principles apply.

**Svelte:** Scoped component styles. Use CSS custom properties for theming.

## Quality Checklist

Before delivering, verify:

- [ ] Custom fonts loaded (not system fonts)
- [ ] Consistent color palette via CSS variables
- [ ] Spacing follows a system (not random values)
- [ ] All interactive elements have hover/focus states
- [ ] Responsive on mobile (test at 375px width)
- [ ] Text is readable (contrast, line-height, max-width)
- [ ] Page has subtle entry animations
- [ ] No default blue links or unstyled form elements
- [ ] Favicon and page title set (if full page)
- [ ] No horizontal scroll on any viewport
